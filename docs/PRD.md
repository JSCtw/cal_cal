## **專案名稱**

LINE飲料熱量機器人

## **目標**

開發一個基於 LINE Messaging API 的聊天機器人，使用者輸入飲料品牌+飲料(+配料)+甜度+冰塊，自動計算出飲品熱量 (與糖量)

## **目標用戶**

- 飲食控制者
- 特定飲料品牌愛好者
    - **飲料品牌清單**：
        - 50嵐
        - Comebuy
        - 一沐日
        - 上宇林
        - 迷客夏
        - 得正
        - 清心福全
        - 麻古茶坊
        - 喫茶趣ToGo
        - 路易莎咖啡
        - 星巴克咖啡
        - 待補上：Cama、UG、可不可
        - 遺珠之憾：大茗、Coco都可、龜記

### 2. 邏輯

- **品牌別名：**
    - **品牌別名對照表**
- **飲品別名**：
    - 在品牌確定後，先比對使用者輸入是否有符合別名，若無，則回飲品名稱欄位尋找。
- **冰量判斷邏輯：**
    
    
    | **使用者輸入** | **查詢對應冰量欄位** | 情境 |
    | --- | --- | --- |
    | 溫 / 熱 | 優先找「冰量 = H」，如果找不到「冰量 = H」，再找「冰量 = I」  | 溫飲 / 熱飲 |
    | 未輸入或「溫」 、 「熱」以外 | 冰量 = I | 以外的情況，包括去冰、常溫，在熱量計算上都以冰飲 (冰量欄位值="I")的為主 |
    
    具體步驟：
    
    1. 使用者輸入包含「溫」或「熱」。
    2. 優先查找冰量 = "H" 的飲料資料。
    3. 如果查不到，再 fallback 去找冰量 = "I" 的資料。
    4. 其他情況（例如「少冰」、「正常冰」、「常溫」）就照常比對冰量欄位。
    
- **配料邏輯：**
    - 使用符號"+"與"-”來判別，”+”則加入配量熱量，"-"則扣除配量熱量，例如去鮮奶油以"-鮮奶油"表示；若加兩種料則重複使用"+”，例如”+珍珠 +粉粿”；若同樣配量加兩份則以"*"表達份數，例如"+珍珠*2”；若無"+"或"-”則判斷為未加配料
- **甜度判斷邏輯：**
    - **甜度設定表**
        
        
        | **甜度** | **公式** | **類別** |
        | --- | --- | --- |
        | 全糖 | 總熱量 | 一般 |
        | 少糖 | 總熱量-總糖量*17.5%*4 | 一般 |
        | 八分糖 | 總熱量-總糖量*20%*4 | 一般 |
        | 半糖 | 總熱量-總糖量*25%*4 | 一般 |
        | 七分糖 | 總熱量-總糖量*30%*4 | 一般 |
        | 微糖 | 總熱量-總糖量*42.5%*4 | 一般 |
        | 三分糖 | 總熱量-總糖量*60%*4 | 一般 |
        | 二分糖 | 總熱量-總糖量*70%*4 | 一般 |
        | 一分糖 | 總熱量-總糖量*80%*4 | 一般 |
        | 無糖 | 總熱量-總糖量*100%*4 | 一般 |
        | 少糖 | 總熱量 | 清心福全 |
        | 半糖 | 總熱量-總糖量*25%*4 | 清心福全 |
        | 微糖 | 總熱量-總糖量*65%*4 | 清心福全 |
        | 無糖 | 總熱量-總糖量*100%*4 | 清心福全 |
    - **若品牌為”清心福全”，則套用類別欄位為”清心福全”的甜度邏輯，若否，則套用類別欄位為”一般”的甜度邏輯**
    - 預設1g的糖是4kcal，所以在甜度比例上糖量皆會*4

- **邏輯順序**
    
    有些邏輯可以寫死在小字典內，以加快查詢速度
    
    ```markdown
    1. 使用者傳訊息，例如：「清心 高山 微糖 熱 大 +珍珠*2」
    
    2. 拆解訊息：
        ➔ 品牌：清心 → 查別名表 → 正式品牌「清心福全」
        ➔ 飲料名稱：高山 → 查正式品牌為「清心福全」的飲料中是否有飲料別名為「高山」 → 飲料名稱「嚴選高山茶」
        ➔ 甜度：正式品牌為「清心福全」，甜度設定表類別為「清心福全」
        ➔ 冰量：熱 → 先找冰量 H，找不到再找冰量 I
        ➔ 配料：+珍珠*2 意味配料表「珍珠」的熱量要*2
        ➔ size：大 → 查別名表 → 尺寸為「L」
    
    3. 在 Drinks 資料表中查詢：
        ➔ 品牌=清心福全 且 飲料名稱=嚴選高山茶 且 size=L 且 冰量=H (冰量=H找不到，故改找同條件但冰量=I)
    
    4. 甜度換算熱量：
        ➔ 甜度設定表類別為「清心福全」且甜度為「微糖」，新熱量 = 熱量（全糖） - 糖量（全糖） × 65% × 4
    
    5. 查 Toppings 表取得珍珠熱量，並*2。
    
    6. 計算總熱量：
        ➔ 飲料甜度換算熱量 + 配料熱量
    
    7. 回覆使用者：
        ➔ 例：「清心 高山 微糖 熱 +珍珠*2，共 453.4 大卡」
    
    ```
    

### 3. LINE Bot 程式

- Chat-GPT建議用 `Python + Flask` + `Google Sheets API` 並部署在  `Railway`
    
    ✅ Python + Flask：
    
    - 本地開發 Web 應用的主體（例如 API server，解析使用者輸入、查詢飲料資料等）。
        - 用 Flask 架構你的小型服務器。
    
    ✅ Railway：
    
    - 是一個免配置的雲端平台（類似 Google Cloud Run）。
    - 可以一鍵部署 Flask App 到線上。
    
    ✅ **費用方面**：
    
    - Railway 有免費方案（約 500 小時/月），適合初期測試。
    - 若使用量大、功能多，會需要升級（約 $5～$20 美元/月）。
    
    ✅ **跟 Google Cloud 的差異：**
    
    | 項目 | Railway | Google Cloud |
    | --- | --- | --- |
    | 容易部署 | ✅ 更適合新手 | ❌ 較為複雜（需要設定 Docker 或 Cloud Build） |
    | 成本 | ✅ 前期免費、簡單定價 | ⚠️ 可能有額外費用如 egress、RAM 等 |
    | 擴充性 | ⚠️ 適合小專案 | ✅ 可支援大型商業部署 |
    | 彈性控制 | ❌ 限於 Railway 提供的模板 | ✅ 可自由整合各項 GCP 服務 |

- 要建立”**初步輸入解析模組**”，包括：
    - 關鍵字對應（品牌/甜度/冰量）
    - 正則拆解語句
    - 模糊比對查詢流程

🔸 增加容錯率：像是這些情況也能解析成功：

- 「五十嵐 珍珠奶茶 正常冰 無糖」
- 「coco 百香綠 半冰 半糖 -椰果」
- 「清心 四季春 無糖 +椰果」

🔸 支援部分缺漏自動補預設值，例如：

- 若只寫「品牌 飲料」→ 自動補冰量、甜度為預設值（正常冰正常糖）。

🔸 防止小錯誤導致「查無資料」

- 多一空格、拼錯品牌（例如「Co co」）能用模糊比對處理。
- 順序錯亂時仍能透過關鍵字拆解。

🔸 評估

| 模組狀態 | 是否推薦 | 適用時機 |
| --- | --- | --- |
| ❌ 不做解析模組 | ✅ 適合開發初期快速驗證 | 僅限內部使用、格式固定 |
| ✅ 建立解析模組 | ✅✅ 適合對外發布、提高使用者容錯 | 中長期發展、商業應用準備 |

可先用固定格式 (`split()` + 預設欄位順序) 上線，接著再逐步加入簡單的輸入解析模組（例如 regex 或關鍵字對應表），這樣前期輕量、後期穩定延伸。

## **非功能需求**

### **1. 性能**

- 支援至少 500名同時訂閱用戶，無明顯延遲。

### **2. 可靠性**

- 若尋找超時或失敗，應跳出提示並於後台記錄錯誤。

### **3. 安全性**

- 使用 LINE Messaging API 的 Channel Access Token 需妥善加密儲存。
- 用戶資料（如 LINE ID）僅用於推送，不得用於其他目的。

### **4. 可擴展性**

- 未來可新增更多品牌或飲品。

### **5. 備援方案**

- 在網頁(如vocus) 架設熱量計算機
